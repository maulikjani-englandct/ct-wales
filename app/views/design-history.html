Design history

Last updated: 2 January 2026

Overview

This prototype explored how to make it easier for people in Wales to view, update and request a review of their council tax band. Earlier versions had two separate journeys — one to update details and one to challenge/review a band. The journey was simplified into a single flow to improve clarity and reduce duplication. From v1.5 onwards, terminology changed from "challenge" to "review" to make it less confrontational for users.

User research guided the design at every stage. We tested with a mix of property owners, renters, landlords, users with low digital confidence, and people with cognitive accessibility needs. Methods included sessions and stepwise highlighter testing to validate journeys and key assumptions.

We tested all prototype elements on smartphones and desktops to ensure usability.

Journey architecture

Core user journeys

We built the service around three journeys:

  1. Journey 1: View property data — Users can see their property's attributes and how the band was calculated
  2. Journey 2: Update property data — A form flow for correcting information we hold about a property
  3. Review Journey — The formal process for requesting a review of a council tax band (called "Challenge" in v1-v1.4)
The journeys are deliberately separate. Users have to view and confirm their property data before they can request a review of their band. This creates a natural decision point and helps prevent inappropriate reviews.

How the structure evolved

The information architecture for the Council Tax Wales Reform prototype journey evolved gradually across versions as user needs and technical capabilities matured:

  • v1.1 to v1.4: The journey used a two-branch approach (/letusknow route) where users could independently choose to update property details or challenge their council tax band.
  • v1.5: Introduced a new /challenge route enabling partial integration of update and review flows, beginning the transition toward a streamlined journey. Changed terminology from "challenge" to "review". Both original and new patterns existed.
  • v1.5.1: Further developed routing logic with /challenge-alt route providing enhanced handling of review and update scenarios.
  • v1.5.2: Completed the journey evolution with streamlined review journey. This fully streamlined the user experience, requiring property details review before band review, thus improving clarity and legal compliance.

This incremental evolution allowed isolated testing of new paths before combining them into a unified journey, improving research clarity and user experience.

Version 1 - Initial prototype (28 July to 7 August 2025)

What we built

  • Simple linear flow: Start → Search → Results → Property Details → Sign in → Extended Details
  • Postcode search: One text input with autocomplete to find addresses
  • Results table: Shows band and local authority, can be sorted
  • Language toggle: Used the HMRC pattern - remembers which language you chose
  • Two levels of detail: Basic property details for anyone, more detailed info once you sign in

Core patterns used

  • Start page: GOV.UK start page pattern with prominent start button
  • Search pattern: Postcode search with autocomplete and results table
  • Summary list: GOV.UK summary list for displaying property data
  • Language toggle: HMRC language selector pattern (Welsh first)

Testing

This version was not formally tested with users - it was an internal build to establish the core patterns and basic structure.

Why we designed it this way

We stuck to proven GOV.UK patterns. The two-tier approach (public and signed-in views) meant we could be transparent about property data while still protecting more sensitive information.

Version 1.1 - Property details update journey (7 to 20 August 2025)

What changed

  • Decision page: Added a "Let us know" page where users choose whether to update data or challenge their band
  • Change links: Each property detail got its own change link in the extended view (GOV.UK summary list pattern)
  • Check answers page: Users review all their changes before submitting (GOV.UK check answers pattern)
  • One question per page: Built 14 separate form pages, each asking about one property attribute (GOV.UK question pages pattern)

New patterns introduced

  • One question per page: GOV.UK question pages pattern - each form asks one specific question (see GOV.UK Design System patterns)
  • Check answers: GOV.UK check answers pattern with change links (see GOV.UK Design System patterns)
  • Conditional reveals: GOV.UK radios with conditional reveals for contextual questions (see Journey splits and conditional routing)

How the journey worked

From the property details page, users clicked a change link, went through the forms one by one (e.g. property type, bedrooms, bathrooms), checked their answers, then got a confirmation.

Each form had a back link and continued to the next form or the check answers page.

Testing

This version was not formally tested with users - it was an internal build to establish the basic journey structure.

Why we designed it this way

Key learning: Users were confusing "updating data" with "challenging the band" - creating a clear decision point was essential to prevent inappropriate submissions.

Why one question per page? This is a core GOV.UK Service Manual pattern. Asking one question per page reduces cognitive load, makes forms work better on mobile, and lets us provide focused help text for each question.

Why check answers? It's a GOV.UK pattern for transactions. Lets users review everything before they submit, which reduces mistakes and makes people more confident. The change links let users go back and fix individual answers without starting over.

Version 1.3 - Property types and authorization (2 September to 19 September 2025)

What we added

  • Self-certification page: GOV.UK checkboxes component where users confirm they're liable for council tax
  • Two-step property type: First pick the main type (house, flat, etc.), then the specific sub-type (using GOV.UK radios with conditional reveals)
  • Split the challenge route: Users choose whether their band is wrong or their property has changed on why challenge page (GOV.UK radios component)
  • Error pages: GOV.UK error page pattern telling users they can't proceed if they're not liable or don't have authority

How the forms worked

  • Property type: Choose house, bungalow, flat, maisonette or other, then get more specific (like studio, 1-bed, 2-bed for flats)
  • Liability check: One checkbox saying "I confirm I am liable for council tax at this property"
  • Conditional forms: Different questions appear depending on why you're challenging

What we learned from testing

Round 1 user research (10-12 participants) revealed several key insights:

  • Understanding council tax: Many users were unclear about how council tax bands are determined, often assuming banding relates to size or number of rooms. They wanted clearer explanations of terms and band purpose
  • Content clarity: Users valued plain language. Welsh speakers preferred Welsh-first content but sometimes referred to English for technical terms. Guidance about which PAD items could be updated was missing or unclear
  • Form questions: Users struggled with complex questions like parking type or property subtypes. They called for more descriptive labels and examples - particularly for parking and area questions which people couldn't answer accurately
  • Inline help: Inline help and details components were more effective than separate glossary pages, which users often ignored
  • Authorization: Clarity needed on who can access or update property data (landlords vs tenants), requiring clearer service guidance

Why we designed it this way

Why two steps for property type? We wanted to keep it simple upfront without overwhelming people, but still capture the detail we need for valuation. Using conditional reveals (GOV.UK pattern) meant we only showed the sub-type question after users selected the main category.

Why use a checkbox? It's the standard GOV.UK pattern for confirmations and declarations. One checkbox with a clear statement is simpler than yes/no radio buttons, and follows GOV.UK guidance for consent and agreement flows.

Version 1.4 - Workflow simplification (11 September to 7 October 2025)
Note: Development of v1.4 began while v1.3 was in user testing, allowing us to iterate quickly based on research findings.

Focus on form simplification

v1.4 kept the two-branch architecture (review property details OR challenge the band) but focused on making the forms simpler and faster to complete.

Making it simpler

  • Shorter parking flow: Cut it from 3 pages down to 2 - removed the dimensions/size page
  • Simplified proxy authorization: Integrated it into the main authorization instead of having a separate page

Pages we deleted

  • parking-size.html (which had 5 form fields)
  • local-authority.html and local-authority-why.html
  • proxy.html
  • errors/no-ata.html

What changed

  • Parking questions now just ask about type (allocated/unallocated) and how many spaces
  • Removed all the complex measurement fields (width, length, area calculations)
  • Reduced the journey from 16 pages down to 11

Testing

Round 2 user research (10-12 participants) tested the simplified forms and validated the removal of complex questions users couldn't accurately answer.

Why we simplified

Key learning: People couldn't estimate parking dimensions accurately - asking for data users don't have creates friction and gives bad data. Only ask for information you really need.
Version 1.5 - Challenge journey and guidance (18 September to 5 November 2025)

Beginning the architecture transition

v1.5 introduced new routing logic that began shifting from the two-branch approach (review OR challenge) toward a streamlined journey where users review property details first, then can challenge. We fine-tuned the routing and conditional logic again in v1.5.1 and v1.5.2 after seeing where people got stuck.

What we added

  • Multi-page guidance: Separate pages for "How bands are decided", "Property details used", "Contact VOA" with a navigation sidebar (content page pattern)
  • Accordions: GOV.UK accordion component used for FAQs and complex guidance that doesn't need to be visible all the time
  • Bands table: GOV.UK table component showing all council tax bands A-I with their 2003 valuation ranges
  • More challenge routes: Split "property changes" into specific scenarios like demolished, split, merged, etc. using GOV.UK radios with conditional routing
  • Inset text: GOV.UK inset text component for important information about the challenge process

Fixes from testing

  • Show full address everywhere: Changed "this property" to the actual address throughout (see Keeping people oriented)
  • Reordered the login: Now goes Property details → Self certification → Government Gateway → Extended details
  • Bigger band display: Made the band letter larger, put supporting info in smaller text

Start page redesign

  • Combined English and Welsh into one page with a table of contents (following GOV.UK start page pattern)
  • Used GOV.UK accordion component for different user needs
  • Explained what the service does and doesn't do before people start (GOV.UK Service Manual guidance)

What we tested

Round 4 user research focused on testing the review journey with 10-12 participants (run in parallel with Round 3 testing v1.5.1). We removed the property details update flow to isolate testing of the new guidance pages and review forms. This let us test:

  • Whether the new guidance helped people understand the challenge process
  • If the challenge routing worked (wrong band vs property changes)
  • How people used the navigation sidebar

Why we designed it this way

Key learning: Saying "this property" caused users to lose track of which property they were viewing. Showing the full address on every page keeps users oriented throughout their journey.

Why multiple guidance pages? Breaking complex information into chunks makes it easier to digest. The sidebar lets people jump straight to what they need. Accordions hide detail until someone wants it.

Why show the full address? If someone's checking multiple properties or looking at old data, saying "this property" meant they lost track of which one they were looking at. The full address keeps them oriented.

Version 1.5.1 - Guidance and start page testing (18 September to 4 November 2025)

Small improvements

  • Better accordions: Clearer visual hierarchy and expand/collapse buttons
  • Help button positioning: Put them inline with labels instead of floating
  • Mobile table: Fixed how the bands table behaves on phones
  • Sign out link: Added Welsh translation and put it in a consistent place

What we tested

Round 3 user research (run in parallel with Round 4 testing v1.5) focused on testing the guidance and property update forms with 10-12 participants. We removed the review journey from this version to isolate testing. This meant we could check:

  • Could people navigate the guidance using the sidebar?
  • Did accordions work for complex information?
  • How did people interpret the bands table?
  • Did showing the full address give enough context?

Why we tested this way

Why test in isolation? The journey is so long that testing everything together would have been overwhelming for participants. Breaking it into separate testing variants (v1.5 for challenge, v1.5.1 for property update) meant each session could focus on a manageable amount of content.

Version 1.5.2 - Complete challenge and review journey (November 2025 - current)

Completed streamlined journey

v1.5.2 completed the architectural transition begun in v1.5. This is the current version of the prototype. It fully implements the streamlined journey where users must review their property details first before they can challenge their band. This was clearer than the earlier two-branch approach: users review details before challenging.

Testing

This version was not user tested - it exists as a reference implementation showing the complete end-to-end journey with all features combined.

What we built

  • Different confirmation pages: GOV.UK confirmation page pattern showing different content depending on whether someone chose formal or informal review
  • Multi-level routing: Challenge reason → Type of property change (8 options) → Different forms depending on what they picked (GOV.UK radios with conditional routing)
  • Autocomplete: Address lookup and contact details with proper autocomplete attributes (following GOV.UK guidance for autofill)
  • Conditional check answers: GOV.UK check answers pattern - the summary page shows/hides sections based on which route someone took
  • 6-month rule: Conditional logic checks if the property changed more than 6 months ago before showing formal review option

8 different property change types

  • Demolished or uninhabitable → Upload evidence + pick a date
  • Part demolished → Evidence and description
  • Split into multiple properties → Form for new property details
  • Merged with other properties → Reference for the merged property
  • Part or all now used for business → What it was before and what it is now
  • Part or all now used as home (domestic use) → Previous and current use
  • Local area changes affecting value → Description and date
  • Something else → Open description field

Language change

Changed "Challenge" to "Review" throughout - in buttons, headings, navigation, confirmation pages. This was a content design decision based on user research findings and our Welsh-first strategy, making the terminology more accessible and less confrontational for users.

Forms we removed

After user research and stakeholder discussions, we removed forms that users couldn't answer accurately or that only applied to a small percentage of properties:

  • floor-level: People couldn't accurately tell if they were on ground, 1st or 2nd floor
  • improvement: Required expert knowledge that users didn't have
  • mixed-use: Only applies to less than 5% of properties - edge cases can be handled elsewhere

Why we designed it this way

Key learning: The streamlined journey (review property details first, then challenge if needed) was clearer than the two-branch approach. Requiring property review before challenge improved clarity and legal compliance.

Why conditional routing? Different property changes need completely different information. Only showing relevant questions makes it faster and less overwhelming.

Why split formal and informal? They're legally different pathways. Formal reviews have statutory timelines and require Notice of Alteration. Informal reviews are quicker but non-binding. The confirmation pages needed to set the right expectations for each.

Why the radical simplification?

We needed to test specific UX patterns without the complexity of the full journey overwhelming participants. By stripping down to just the concepts being tested, we could get clearer feedback on what worked and what didn't.

2. Property Display Patterns (PADs)

We tested 4 different display patterns:

Based on patterns from previous testing rounds, we'll explore:

  • PADs Presentation: Adding comparable properties, calculation tools, clearer explanations
  • Journey Reordering: Moving scenario selection earlier in the flow

Why we designed it this way

Why integrate evidence upload here? Positioning it after explaining why the band is wrong creates a logical flow - users explain their reasoning, then can optionally provide supporting evidence, then specify which band they believe is correct.

Why multi-page evidence? The decision point before file upload follows progressive disclosure principles - only show the upload interface when users confirm they have documents to share.

Key themes

Journey splits and conditional routing

Started with one linear path, ended up with multiple routes based on what users needed:

  • First split: Update property data or review band (called "challenge" in v1-v1.4)
  • Second split: Wrong band or property has changed
  • Third split: 8 different property change types, each with different forms
  • Fourth split: Formal or informal review

Each decision point meant we only showed people what was relevant to their situation.

GOV.UK Design System patterns

All design decisions were based on GOV.UK Design System components and Service Manual patterns:

  • Question pages (one thing per page): GOV.UK Service Manual pattern - every form asks one question to reduce cognitive load
  • Check answers pattern: GOV.UK pattern for transactions - standard summary with change links and conditional sections
  • Radios with conditional reveals: GOV.UK component for property type, challenge reason, showing/hiding follow-up questions
  • Details component: GOV.UK component for extra help without cluttering the page
  • Summary list component: GOV.UK component for showing property details with change links
  • Back links: GOV.UK back link component on every form page

Keeping people oriented

Design choices to stop people getting lost (following GOV.UK guidance on navigation):

  • Show the full address on every page (not "this property") - helps users maintain context
  • Back links: GOV.UK back link component on all forms
  • Change links: GOV.UK check answers pattern - keep you in context when editing from check answers
  • Breadcrumbs: GOV.UK breadcrumbs component in the guidance section
  • Language toggle: HMRC language selector always in the header

Built-in help instead of glossaries

Started with separate glossary pages, ended with help built into the forms (following GOV.UK guidance on contextual help):

  • Hint text: GOV.UK hint text on all form inputs explaining what we need
  • Details component: GOV.UK details component for longer explanations that don't clutter the page
  • Inset text: GOV.UK inset text component for highlighting important information
  • Guidance within the journey: Not separate reference pages - following GOV.UK Service Manual guidance to provide help in context

Testing showed people rarely clicked through to separate glossaries - they were more likely to read help that was right there on the page.

Welsh-first bilingual design

The service provided a fully bilingual experience using the HMRC Welsh language toggle pattern:

  • Language choice: Users can select English or Welsh; their preference is saved in a session cookie and remembered throughout the journey
  • URL parameter: Language preference is visible in the URL (?languagePreference=en or cy) for easier testing and sharing
  • Seamless switching: Users can switch language mid-journey without losing form data
  • Consistent placement: Toggle always visible in the header, following HMRC pattern guidance

Design process and technical implementation

Two-week sprint cycle: Test → Learn → Update → Test again. This rhythm kept research and design tightly integrated throughout the project.

Team structure

The UCD team consisted of:

  • 1 User Research Lead
  • 2 User Researchers
  • 1 Service Designer
  • 1 Content Designer
  • 1 Interaction Designer

The Content Designer worked with a separate trio writing team (content designer, translator, SMEs) to produce bilingual content.

Bilingual content workflow

Content was drafted in English first, translated into Welsh, then refined in English to ensure consistency across languages. This workflow ensured both languages remained aligned throughout the prototype.

During testing, users could interact with English and Welsh content using the HMRC Welsh language toggle (see Key themes → Welsh-first bilingual design for toggle pattern and technical implementation). This approach allowed realistic usability testing without duplicating journeys and made iteration easier.

How the process worked: from research to design

Every two weeks, the prototype was tested, findings were gathered, updates were made to screens and content, and then tested again.

The workflow

  1. User researchers shared insights from each round (about 10 participants per cycle)
  2. Interaction designer produced a first draft of the prototype based on those findings with placeholder or current content
  3. Trio writing team (content designer, translator, subject expert) wrote bilingual content
  4. Interaction designer integrated content into the prototype and built any new logic
  5. Whole UCD team reviewed updates before the next testing round

Major terminology decisions (like changing "Challenge" to "Review") were agreed collectively with the UCD team to ensure alignment between design, content, and service design.

Prototyping tools and approach

GOV.UK Prototype Kit (runs on Express.js) was the main tool.

What it let us build

  • Realistic prototypes: Mid to high fidelity that felt like a real service
  • Language switching: Full English and Welsh with persistent preference
  • Session persistence: Form data that persists as users navigate

What we tested

Each round focused on specific aspects of the user journey:

  • Journey routing: Decision points and how the conditional logic worked
  • Core journeys: Update property data and challenge band paths
  • Check answers: Conditional sections based on user choices

How we measured success

  • Could people complete the task without getting stuck?
  • What did people say in testing sessions?
  • Did users understand the terminology and guidance?
